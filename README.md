## 📌 Overview

This repository is a **Personal Knowledge Management (PKM) system** created for the Software Engineering course.

It is not just a place for storing notes. It is a structured environment for:

- tracking how my understanding evolves  
- documenting technical and product decisions  
- running experiments and analyzing outcomes  
- improving how I design and build software systems  

This repository supports the development of:

🥗 **Smart Nutrition & Fitness Tracker (AI-powered)**

and focuses on the *thinking process behind building the system*, not just the final implementation.

---

## 🎯 Purpose

Most note-taking systems focus on collecting information.

This repository focuses on:

- structured thinking in software engineering  
- decision traceability (why something was built a certain way)  
- versioned learning using Git  
- handling uncertainty in real development  
- continuous improvement during the project lifecycle  

---

## 🧠 Core Principles

### 1. Separate thinking layers

Different types of knowledge are stored separately:

- **Beliefs** → assumptions and mental models  
- **Decisions** → choices made under uncertainty  
- **Experiments** → implementations and results  

This avoids mixing opinions, actions, and outcomes.

---

### 2. Write for future understanding

Each entry must be understandable later without additional context.

Notes should always include:
- problem  
- reasoning  
- outcome  

---

### 3. Use Git as a learning system

Git is used not only for code, but also for:

- tracking evolution of ideas  
- documenting changes in understanding  

---

### 4. Be explicit about uncertainty

Each entry should clearly indicate:

- hypothesis  
- observation  
- decision  
- result  
- open questions  

---

### 5. Prefer synthesis over accumulation

The goal is not to collect notes, but to:

- connect ideas  
- identify patterns  
- improve decision-making

## 📁 Folder Description

### 📥 `inbox/`
Raw, unprocessed inputs:
- lecture notes  
- quick ideas  
- observations  

---

### 🧩 `beliefs/`
Current assumptions about:

- nutrition tracking accuracy  
- AI-based food analysis reliability  
- user behavior in fitness apps  

Example:
- "Users prefer fast logging over accurate logging"

---

### ⚖️ `decisions/`
Documented decisions such as:

- choosing FastAPI vs Node.js  
- selecting database (PostgreSQL vs Supabase)  
- defining MVP scope  

Each decision includes:
- context  
- alternatives  
- reasoning  

---

### 🧪 `experiments/`
Logs of:

- AI food analysis tests  
- feature implementations  
- performance experiments  
- failures and lessons learned  

---

### 🧠 `synthesis/`
Higher-level summaries:

- weekly reflections  
- patterns and insights  
- changes in understanding  

---

### 📚 `resources/`
Useful materials:

- articles  
- documentation  
- APIs (OpenAI, Open Food Facts)  

---

### 📄 `docs/`
General course notes and supporting documentation.

---

## 🔄 Workflow

1. Capture ideas in `inbox/`  
2. Convert into structured entries (belief / decision / experiment)  
3. Apply ideas in the Smart Nutrition Tracker project  
4. Analyze outcomes  
5. Summarize insights in `synthesis/`
